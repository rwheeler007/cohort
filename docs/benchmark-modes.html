Cursor-based pagination has well-established precedent across high-scale APIs -- Twitter's v2 API, Stripe, Slack, and the GitHub GraphQL API all use keyset/cursor patterns precisely because offset pagination degrades to O(n) as the offset grows, since the database must scan and discard all preceding rows. The seminal work here is Markus Winand's "Use The Index, Luke" which demonstrates how WHERE id > :last_seen ORDER BY id LIMIT :size leverages index seeks for constant-time page fetches regardless of depth. For our use case, I'd recommend the opaque cursor approach (base64-encoded keyset values) over raw column values, since it decouples the client from internal sort implementation and lets us change the underlying ordering strategy without breaking the API contract.
Claude-only 122 in / 228 out | 13.0s
The three dominant patterns in production APIs are offset/limit (GitHub REST, Stripe), cursor-based (Facebook Graph API, Slack, Relay spec), and keyset pagination (used internally by many databases). Offset/limit is simplest but degrades on large tables due to OFFSET N requiring scanning N rows; cursor-based avoids this by encoding an opaque position token, making it O(1) for indexed columns. Stripe's approach is notable -- they use cursor-based with starting_after=obj_id and limit, which maps cleanly to a WHERE id > :cursor ORDER BY id LIMIT :n query and avoids the stale-data issues of offset pagination when records are inserted or deleted mid-traversal. For a /users endpoint, I'd recommend cursor-based with a stable sort key (like created_at + id composite) since it scales well and is the clear industry consensus for any dataset that could grow beyond a few thousand rows.

RS Researcher
Smarter 447 in / 93 out | 1.2s
This design aligns well with the Twitter API v2 and GitHub REST API patterns, which use opaque tokens to prevent user enumeration and ensure consistent ordering. To maintain backward compatibility, we should include a limit parameter with a default value of 100 for existing clients while deprecating the offset approach in our documentation. I recommend validating that the cursor token is strictly alphanumeric to avoid injection risks before passing it to the database layer.
Smartest 2,808 in / 2,196 out | 35.8s
Cursor-based pagination using the limit + 1 fetch pattern is well-established -- Stripe, GitHub, and Slack all use opaque cursor tokens over offset-based approaches precisely because offsets break under concurrent inserts/deletes. The critical migration concern is that switching /users from unbounded results to a default limit of 100 is a breaking change for any client iterating without pagination; a Deprecation header plus a temporary ?legacy=true escape hatch gives existing consumers a migration window. On the cursor token itself, base64-encoding the primary key is common but you should HMAC-sign it server-side to prevent clients from tampering with the cursor state or using it as an injection vector. Finally, make sure the final-page logic handles results.length <= limit correctly -- if you always slice off the extra record without checking, you'll return has_more: true on the last page and clients will loop forever.
Claude-only 548 in / 237 out | 12.9s
The backward compatibility concern is real -- any existing clients calling GET /users without pagination params will go from getting all records to getting only the first 25. Stripe handled this well: they kept the unpaginated response shape identical (still returns a list) but added has_more and next_cursor as new fields, so existing clients still work but only see the first page. I'd also flag that cursor-based pagination has a known limitation for prior art: it doesn't support "jump to page N" which offset/limit does, but in practice almost no production API consumer actually needs random page access -- infinite scroll and "load more" patterns dominate. One historical note worth considering: both Twitter's v2 API and Shopify's GraphQL API migrated from offset to cursor-based and documented the transition as non-breaking by defaulting the first page when no cursor is provided, which is exactly the pattern the architect proposed.

Critical edge cases to validate: (1) cursor tampering -- verify that a valid base64 string containing a SQL injection payload like 2024-01-01T00:00:00; DROP TABLE users--:1 is safely rejected by the datetime parser rather than passed to the query, (2) boundary conditions -- empty table returns {"data": [], "next_cursor": null, "has_more": false}, exactly limit+1 rows returns has_more: true with only limit items in data, and a cursor pointing to a deleted record still works correctly since the > comparison skips over gaps, (3) the base64 cursor leaks created_at timestamps which could be an information disclosure concern -- consider HMAC-signing or encrypting the cursor so clients can't decode user creation times or forge cursors to enumerate records. RS Researcher
Smarter 787 in / 88 out | 1.2s
Your security review is spot on; rate limiting is critical since cursor enumeration can bypass pagination limits even with signed tokens, as seen in vulnerabilities affecting the Stripe API. For Fernet key rotation, ensure you implement a dual-key strategy where old keys remain valid for a grace period to prevent breaking existing sessions during updates. Smartest 1,742 in / 694 out | 22.4s
The limit + 1 off-by-one pattern causing infinite loops is a well-documented class of pagination bugs -- cursor-based systems that fetch limit + 1 to detect "has next page" must still only return limit rows to the caller, or the cursor never advances. For HMAC signature verification, hmac.compare_digest isn't optional -- Python's == on strings short-circuits on the first differing byte, leaking timing information that lets an attacker reconstruct valid signatures incrementally. The dual-key rotation strategy mirrors how production systems like AWS KMS and Stripe handle key rollover: accept signatures from both old and new keys during a grace window, then hard-cut to invalidate the old key. On the index side, databases can drop an index scan when the ORDER BY columns don't match the index's sort order -- explicitly aligning them and using EXPLAIN ANALYZE under load keeps pagination queries on the fast path.
Claude-only 1,204 in / 202 out | 12.9s
The tester's point about HMAC-signing cursors has strong prior art -- Slack's API uses signed cursors precisely to prevent enumeration attacks, and Facebook's Graph API encrypts theirs entirely. However, I'd push back on making this a launch blocker: Stripe and GitHub both ship plain base64 cursors with a documented "cursors are opaque and may change format" contract, treating cursor forgery as a low-severity issue since the underlying query is parameterized and can't return data the caller isn't already authorized to see. The pragmatic move is to ship with unsigned cursors now, add a cursor_version prefix byte (e.g., v1:base64payload) so you can transparently migrate to HMAC-signed cursors later without breaking clients, which is exactly how Shopify handled their cursor format evolution.

Why the Condenser Exists

The test above used short system prompts (~170 tokens). In production, every agent carries its full context: personality, principles, capabilities, persistent memory, learned facts from RSS feeds, and channel conversation history. That's a very different prompt.

[!] What happens without distillation

We ran the same scenario through the full production pipeline. The 9B local model received the complete agent prompt -- personality, principles, capabilities, persistent memory, RSS-sourced facts, collaboration rules, and channel context. It looped. The model regurgitated its own system prompt text repeatedly, interspersed with Chinese characters meaning "stuck in a loop." Zero usable output from ~16K tokens of input.

This isn't a model quality problem -- it's a context management problem. A 9B model has the knowledge to answer well (it scored 6.3/10 with short prompts), but drowning it in 4,000+ tokens of system context before the actual question causes prompt-following collapse.

[*] What the Smartest pipeline actually does

Smartest mode solves this with a three-phase pipeline. Phase 1: Qwen processes the full context with its 256K window and reasons about the answer (thinking tokens, free). Phase 2: Qwen distills its own reasoning into a structured briefing (~200-400 tokens of pure signal). Phase 3: Claude receives that tight briefing instead of the raw 4,000+ token prompt.

The result: Claude gets better input than reading raw context. The distillation extracts key findings, recommended approaches, and concrete details -- pre-digested by a model that already did the hard work of synthesizing the agent's full knowledge base. That's why Smartest sent only 3,348 tokens to Claude across all 8 responses, compared to Claude-only's 8,046 tokens -- 58% fewer paid tokens with comparable quality.

Full agent context (4,000+ tokens) --[Qwen reasoning, free]--> Distilled briefing (~300 tokens) --[Claude, paid]--> Final response

The Real Story: You Don't Choose One

In production, Cohort runs Smarter for 95% of agent coordination -- free, fast, good enough. When a message genuinely needs premium quality (a security review, a critical architecture decision), you escalate that one message to Smartest. You pay $0.004 for one Claude call instead of $0.042 for eight.

That's the tiered architecture: local model handles the volume, Claude handles the exceptions. Your GPU does 95% of the thinking. Your wallet barely notices the other 5%.
